# SafeStay Network — Backend V1

Production-grade dual-portal backend for a government-grade hotel management and law enforcement surveillance system. Built with Node.js 20, TypeScript (strict), Express, PostgreSQL, Prisma, Redis, and JWT auth.

---

## Architecture Overview

SafeStay enforces a strict **Abstraction Wall** between two completely separate portals:

| Portal | Purpose | Auth |
|--------|---------|------|
| **Hotel Portal** | Room management, guest check-in/check-out, ledger | JWT with `portalType: HOTEL` |
| **Police Portal** | Criminal profiles, verification engine, match alerts, dashboard | JWT with `portalType: POLICE` + 14-level RBAC + jurisdiction filtering |

The hotel portal is entirely ignorant that police surveillance exists. Hotel API responses, error messages, and code paths contain zero references to police, criminal, or verification concepts (Abstraction Wall — Skill 6).

## Database Schema

### Jurisdiction Hierarchy (5-tier)
```
State → Zone → Range → District → Station
```
Each level stores a `jurisdictionPath` string (e.g., `"s1/z2/r3/d4/st5"`) enabling fast row-level security filtering using `startsWith`.

### Core Models
| Model | Domain | Notes |
|-------|--------|-------|
| `Hotel` | Hotel | `nearestStationId` and `jurisdictionPath` are backend-only fields, never returned to hotel frontend |
| `Room` | Hotel | Auto-generated on hotel registration |
| `Guest` | Hotel (write) / Police (read) | Photos mandatory for check-in; written by hotel, read-only by police via verification |
| `PoliceUser` | Police | Linked to a `Station`; `jurisdictionPath` copied at creation |
| `PoliceRank` | Police | 14 levels (seeded); Level 1 = DGP, Level 14 = Constable |
| `CriminalProfile` | Police | Jurisdiction-scoped; `aadhaarNumber` stored raw in V1 (encryption in V1.5) |
| `MatchAlert` | Verification | Generated by verification engine when score ≥ 0.40 |
| `AuditLog` | Both | Every write on sensitive resources is logged here |

## Setup & Running

### 1. Prerequisites
- Node.js 20 LTS
- PostgreSQL 16
- Redis 7

### 2. Install dependencies
```bash
cd safestay-backend
npm install
```

### 3. Configure environment
```bash
cp .env.example .env
# Edit .env with your DB credentials and secrets
```

Generate secure secrets:
```bash
node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"
```

### 4. Run database migration
```bash
npx prisma migrate dev --name init_safestay_v1
npx prisma generate
```

### 5. Seed police ranks
```bash
npm run seed
```

### 6. Start development server
```bash
npm run dev
# Server starts at http://localhost:4000
# API docs at http://localhost:4000/api-docs
# Health check: GET http://localhost:4000/health
```

### 7. Run tests
```bash
npm test
# or with coverage:
npm run test:coverage
```

### 8. Production (PM2)
```bash
npm run build
pm2 start dist/index.js --name safestay-backend
```

## Security Architecture

### Authentication
- **Access tokens**: JWT, 15-minute expiry, contain `sub`, `portalType`, `rankLevel` (police), `jurisdictionPath` (police)
- **Refresh tokens**: 128-byte random hex, stored in DB, revocable individually
- **Password hashing**: Argon2id

### Row-Level Security (Jurisdiction)
Every police-context Prisma query includes:
```typescript
where: { jurisdictionPath: { startsWith: req.jurisdictionPath } }
```
An officer at `state1/zone1/range1/district1/station1` can only see data within that station. A DGP at `state1` sees the entire state.

### RBAC (14 Levels)
| Level | Rank | Key Permissions |
|-------|------|----------------|
| 1 | DGP | Full state access |
| 6 | SP | Create officers, full district |
| 8 | DSP/CI | Soft-delete criminals |
| 9 | Inspector | Station-level; can trigger verification with HC |
| 10 | SI | Review/dismiss match alerts |
| 12 | Head Constable | Create/update criminal profiles, trigger verification |
| 13-14 | Senior Constable / Constable | Read-only |

### Security Headers
All responses include: `HSTS`, `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY`, `X-XSS-Protection`, `Referrer-Policy: no-referrer`, `Permissions-Policy`.

### Rate Limiting
- Global: 100 requests / 15 min / IP
- Auth endpoints: 10 requests / 15 min / IP

### Abstraction Wall
The hotel portal is completely unaware that police surveillance exists:
- Hotel routes use only `authenticate + authorizeHotel`
- Hotel responses never contain: `police`, `criminal`, `verification`, `matching`, `alert`, `surveillance`, `nearestStation`, `jurisdictionPath`
- CORS whitelists hotel and police frontends separately

## Verification Engine (Part 9)

The manual verification engine (`src/services/verification.service.ts`) runs a cross-product match between all WANTED/ABSCONDING criminals and all currently checked-in hotel guests within the officer's jurisdiction.

### Match Scoring Weights (V1 — Exact Match Only)
| Field | Weight | Notes |
|-------|--------|-------|
| Aadhaar | 0.55 | Strongest identifier |
| Name | 0.20 | Normalized: lowercase, alphanumeric only |
| Phone | 0.15 | Exact match against criminals.phones array |
| Age | 0.05 | Within ±3 years |
| Passport | 0.05 | Exact match (normalized) |

- **Threshold**: Scores ≥ 0.40 create a `MatchAlert` (PENDING_REVIEW)
- **Top matches**: Scores ≥ 0.70 are returned in the API response immediately
- **Deduplication**: Existing PENDING_REVIEW alerts for the same pair are updated (not duplicated)

### V2 Planned Improvements
- Soundex / Fuzzy name matching
- Cron-based sweep verification
- Auto-dispatch on new criminal profile creation (INJECTION trigger)
- WhatsApp / Email alert dispatch for confirmed matches

## Audit Logging

Every sensitive write operation records an `AuditLog` entry containing:
- `actorId` + `actorType` (HOTEL or POLICE)
- `action` (CREATE, UPDATE, DELETE, LOGIN, LOGOUT, EXPORT, VERIFICATION_RUN)
- `resourceType` + `resourceId`
- `ipAddress` + `userAgent`

Operations audited: criminal profile CRUD, police officer CRUD, verification runs, match alert reviews, logins/logouts.

## File Uploads

- **Guest check-in**: Requires `guestPhoto` (face) + `idDocument` (ID card) — both mandatory. International guests additionally require `formC`.
- **Criminal profiles**: Optional `photo` field.
- Files stored under `./uploads/guests/` and `./uploads/criminals/` with randomized filenames.
- Allowed types: JPEG, PNG, WEBP. Max size: configurable via `MAX_FILE_SIZE_MB` (default 10MB).
- **Only relative paths** are stored in the DB — absolute server paths are never exposed in API responses.

## Known Issues & Notes for V1.5 / V2

> **BUG-01 — FIXED (April 2026):** `AuditLog→PoliceUser` Prisma FK relation removed. `actorId` is now a plain `String` with no FK constraint. Manual actor lookups replace `prisma include`. See schema.prisma AuditLog model.

> **BUG-02 — FIXED (April 2026):** `jest.config.ts` corrected from invalid `setupFilesAfterFramework` to `setupFilesAfterEnv`. `tests/setup.ts` created with global `clearAllMocks` / `restoreAllMocks` hooks.

> **HARD-01 — FIXED (April 2026):** `JWT_REFRESH_SECRET` is now wired. `verifyRefreshToken()` and `generateRefreshJwt()` added to `token.service.ts`. Access and refresh signing keys are now fully isolated.

1. **Aadhaar stored raw**: `aadhaarNumber` on `Guest` and `CriminalProfile` is stored in plaintext in V1. Field-level encryption (AES-256-GCM) must be added before production deployment. V1.5 task — do NOT implement before testing is complete.

2. **No fuzzy name matching**: V1 uses exact normalized string comparison. Soundex or Levenshtein matching is scheduled for V2.

3. **No cron sweep**: Verification must be manually triggered via `POST /api/v1/verification/run`. Automated periodic sweeps are V2.

4. **No injection trigger**: In V2, saving a new `CriminalProfile` will automatically run verification against all active hotel guests in jurisdiction.

5. **Redis not used in V1 core flows**: Redis is connected but V1 does not use it for session storage or caching. It is scaffolded for V2 (rate-limit state, verification job queue).

6. **No OCR / Form C auto-generation**: Manual document upload only in V1.

7. **No CSV/PDF export**: Scheduled for V1.5 as a separate task.

8. **Test environment**: Tests use mocked Prisma to avoid requiring a live database. For integration tests, configure a separate `safestay_test` database and set `DATABASE_URL` in a `.env.test` file.
